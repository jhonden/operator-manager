## 一、背景与起点

### 1.1 项目介绍

Operator Manager 是一个基于算子（Operator）和算子包（Operator Package）的业务流程管理平台，支持算子的创建、配置、打包、发布和执行。

**技术栈：**
- **后端**：Spring Boot 3.2.x + Java 21 LTS
- **前端**：React 18 + TypeScript 5.x + Vite 5.x + Ant Design 5.x
- **数据库**：PostgreSQL 15+
- **缓存**：Redis 7.x
- **存储**：MinIO（S3 兼容）
- **认证**：JWT + Spring Security

**核心功能模块：**
- 算子管理：算子的 CRUD、参数管理、代码编辑、业务逻辑编辑
- 算子包管理：算子包 CRUD、算子组合、公共库同步、打包配置
- 公共库管理：公共库 CRUD、文件管理、代码编辑、分类

**开源地址**：https://github.com/jhonden/operator-manager

## 二、协作初期的挑战

### 2.1 认知对齐困难

**问题表现：**

1. **技术栈理解的偏差**
   - 我需要理解现有的技术选择和架构设计
   - 对某些技术选型的决策背景不清楚
   - 导致在实现时可能偏离原有设计思路

2. **业务理解的不一致**
   - 对同一个需求的理解存在差异
   - 导致实现结果与预期不符
   - 需要频繁沟通和返工

3. **开发节奏的不匹配**
   - 双方的工作习惯不同
   - 对任务优先级的判断不一致
   - 导致协作效率不高

**博客案例：第1篇 - 方案设计文档与代码实现差异**

在开发早期，我们发现了一个重要问题：当系统的方案设计文档与代码当前的实现已存在很大差异时，该怎么办？

当时的情况是：
- 前期的设计方案与实际代码实现差异较大
- 新成员难以理解"为什么这么做"
- 导致开发方向不清晰

**讨论和解决：**
1. 明确文档与代码的关系：文档是设计蓝图，代码是具体实现
2. 当两者差异较大时，优先遵循代码的实际实现
3. 及时更新文档，使其与代码保持一致
4. 记录差异原因和决策过程，便于后续理解

这个讨论帮助我们建立了第一个协作默契：**"遇到差异，先看代码，再对照文档"**。

**博客链接**：[当系统的方案设计文档与代码当前的实现已存在很大差异时该怎么办](https://www.cnblogs.com/jianxin-blog/p/19624150)

### 2.2 规范逐步形成的过程

**问题背景：**

在开发初期，我们遇到了各种问题，每次解决一个问题，就讨论并形成一个约定。但这些约定都是零散的、口头式的，没有形成体系。

**第2-3天：从零散约定到初步规范**

- **遇到的第一个问题**：前后端开发顺序
  - 我先写前端代码，然后发现后端 API 还没实现
  - 导致频繁返工

  - **形成约定**：先后端再前端

- **遇到的第二个问题**：代码提交规范
  - 没有明确提交标准
  - 导致提交信息混乱、无法追溯

  - **形成约定**：修改后必须验证，验证通过才能提交

- **遇到的第三个问题**：文档注释语言
  - 代码注释不统一
  - 中英文混用

  - **形成约定**：文档和注释使用中文，代码日志使用英文

**博客案例：第3篇 - 构建项目规约**

在第3天，我记录了构建项目规约的过程，这是我们系统化规范的起点。

当时整理的内容包括：
1. 文档和注释语言规范
2. 后端开发规范（编码、异常处理、日志、事务）
3. 前端开发规范（TypeScript、组件、样式）
4. 测试规范
5. 安全规范

**博客链接**：[构建项目规约](https://www.cnblogs.com/jianxin-blog/p/19626689)

### 2.3 协作节奏的磨合

**从"频繁询问"到"按流程执行"**

**第1-2天：频繁询问阶段**

- 每做一件事都要问："这个怎么做"、"用哪个框架"
- 对方需要反复解释
- 双方的效率都不高

**第3天开始：按流程执行阶段**

- 开始有了初步的规范文档
- 遇到问题先查文档
- 文档没写再讨论
- 协作效率显著提升

**默契的形成：**

1. **"规范"的含义**
   - "规范"不只是一份文档
   - 意味着"先查文档，再问问题"
   - 避免重复讨论

2. **决策流程的标准化**
   - 大决策：讨论并记录
   - 小决策：按规范执行
   - 减少沟通成本

3. **文档查阅的习惯**
   - 遇到问题先查文档
   - 文档不清晰再讨论
   - 讨论结果更新文档

**效率变化：**
- 第1天：频繁询问，效率较低
- 第2天：初步规范，效率有所提升
- 第3天后：按流程执行，协作顺畅
- 第5-8天：默契形成，高效协作

## 三、规范的逐步沉淀

### 3.0 核心机制：双文档体系设计

**问题的发现：**

在开发过程中，我们发现了一个重要问题：需求设计文档只能反映"计划做什么"，无法承载"实际怎么做的"。这导致：

1. 新成员难以理解整体设计
2. 设计变更没有记录
3. 代码与设计文档不同步
4. 知识分散，难以检索

**博客案例：第10篇 - 构建模块功能设计文档**

在第8天，我记录了构建模块功能设计文档的思考过程，这是双文档机制设计的核心。

**双文档体系的设计思路：**

```
需求设计文档                          模块功能设计文档
├── 阶段：开发前                      ├── 阶段：开发后 + 持续维护
├── 关注点：需求分析、设计方案          ├── 关注点：核心功能、设计实现
├── 精度：按需求增量新增              ├── 精度：按模块整体承载
├── 状态：设计稿                      ├── 状态：系统蓝图
├── 举例：2026-02-20-算子基本信息扩展-需求设计.md
└── 位置：docs/requirements/            └── 位置：docs/modules/
```

**两种文档的互补关系：**

| 对比项 | 需求设计文档 | 模块功能设计文档 |
|--------|-------------|--------------|
| 回答的问题 | "为什么这么做？" | "实际怎么做的？" |
| 时机 | 开发前创建 | 开发后创建 + 持续维护 |
| 关注点 | 需求分析、设计方案 | 核心功能、设计实现 |
| 精度 | 按需求增量（片段式） | 按模块整体（完整式） |
| 状态 | 设计稿（不变） | 系统蓝图（持续更新） |
| 命名示例 | `YYYY-MM-DD-XXX-需求设计.md` | `operator-management.md` |
| 目录位置 | `docs/requirements/` | `docs/modules/` |

**博客链接**：[构建模块功能设计文档](https://www.cnblogs.com/jianxin-blog/p/19629633)

### 3.1 需求设计阶段

**遇到的问题：**

1. 需求频繁变更
2. 需求边界不清晰
3. 设计讨论效率不高

**博客案例：第2篇 - 增量需求开发**

   - 每个阶段都要明确确认

3. **文档化决策**
   - 所有讨论结果记录在文档中
   - 便于后续追溯

### 3.3 服务管理与验证阶段

**遇到的问题：**

1. 服务启动混乱
2. 验证缺失
3. 修改后直接提交（未验证）

**形成的关键机制：**

1. **服务启动规范化**
   - 整体启动脚本：start-all.sh / start-all.bat
   - 单独启动脚本：start-frontend.sh / start-backend-local.sh / start-backend.bat
   - 支持本地模式和 Docker 模式

2. **服务停止规范化**
   - 整体停止脚本：stop-all.sh / stop-all.bat
   - 单独停止脚本：stop-frontend.sh / stop-backend-local.sh / stop-backend.bat

3. **验证流程标准化**
   - 后端验证：编译 → 启动 → 功能测试
   - 前端验证：编译 → 启动 → 功能测试
   - 验证通过标准

4. **提交流程标准化**
   - 验证完成 → 用户确认 → 提交
   - 提交信息规范：中文、类型、简短描述

**形成的完整规范：**

2. **[代码提交流程](https://github.com/jhonden/operator-manager/blob/main/docs/standards/code-submission-workflow.md)**

### 6.3 给其他团队的建议

#### 关于博客写作

1. **不要等到项目结束再总结**
   - 开发过程中就记录
   - 每次遇到问题、做出决策都写一篇
   - 实时博客比事后总结更真实、更有价值

2. **记录当时的思考过程**
   - 不仅要记录"做了什么"
   - 更要记录"当时是怎么思考的"
   - 保留决策的背景和依据

3. **问题解决要详细**
   - 遇到的问题要详细描述
   - 解决方案要具体可行
   - 便于后续遇到类似问题时参考

#### 关于规范沉淀

1. **先形成规范，再严格执行**
   - 规范应该来自实际问题和讨论
   - 不是凭空想象或照搬其他团队
   - 形成后严格遵守，不要随意打破

2. **持续更新和优化**
   - 规范不是一次性的
   - 要根据实际情况持续更新
   - 发现不合理的地方及时讨论修改

3. **消除重复，明确分工**
   - 文档间避免重复内容
   - 每个文档有明确的职责边界
   - 通过交叉引用形成文档网络

#### 关于协作

1. **早期多沟通，建立共同语言**
   - 开始阶段多沟通
   - 对关键术语达成共识
   - 建立共同的决策流程

2. **及时记录决策，避免反复讨论**
   - 每次决策都记录
   - 后续不再讨论同样的问题
   - 形成"查文档"的默契

3. **培养"查文档"的习惯**
   - 遇到问题先查文档
   - 文档不清晰再讨论
   - 讨论结果更新文档

#### 关于文档体系

1. **采用双文档体系**
   - 需求设计文档：记录"为什么"
   - 模块设计文档：记录"怎么做"
   - 形成从设计到实现的完整链条

2. **代码改动同步更新文档**
   - 完成功能先更新文档再提交代码
   - 规范中明确要求
   - 保持文档与代码同步

3. **建立文档索引**
   - 总入口文档（如 CLAUDE.md）
   - 规范文档索引（如 README.md）
   - 模块文档索引（如 system-overview.md）
   - 便于快速导航和查找

### 7.2 核心学习价值点

#### Vibe Coding 模式的价值

1. **如何快速建立协作默契**
   - 从频繁询问到按流程执行
   - 建立"查文档"的默契
   - 减少沟通成本

2. **如何在短期沉淀可复用的协作体系**
   - 从零散约定到系统化文档
   - 从口头规范到文档化规范
   - 从单一文档到完整体系

3. **从开发驱动到文档驱动**
   - 不是为文档而文档
   - 而是在解决问题中沉淀规范
   - 规范来自实际需求，不是凭空想象

#### 规范驱动开发的价值

1. **从口头约定到文档化规范**
   - 明确的语言使用规范
   - 清晰的编码规范
   - 标准化的提交流程
   - 系统化的服务管理

2. **消除文档重复，明确分工**
   - 每个文档有明确的职责
   - 通过交叉引用形成文档网络
   - 便于快速找到需要的内容

3. **持续更新和优化**
   - 规范不是一成不变的
   - 根据实际情况持续优化
   - 形成持续改进的机制

#### 双文档体系的价值

1. **需求设计文档：回答"为什么"**
   - 记录设计决策的原因和背景
   - 便于追溯和理解

2. **模块设计文档：回答"怎么做"**
   - 记录实际的实现方式
   - 快速上手和理解核心设计

3. **需求与模块的互补**
   - 从需求到设计的完整链条
   - 新成员理解决策和实现
   - 设计变更时的同步机制

#### 实时博客记录的价值

1. **过程记录**
   - 捕捉当时的思考过程
   - 记录遇到的困难和解决方案

2. **决策依据**
   - 保存决策的背景和理由
   - 便于后续理解"为什么这么做"

3. **知识网络**
   - 形成可检索的知识库
   - 提供复盘和学习的素材
