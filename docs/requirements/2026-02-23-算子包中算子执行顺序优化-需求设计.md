# 算子包中算子执行顺序优化需求设计

> **创建日期**：2026-02-23
> **需求状态**：待确认
> **版本**：v1.0

---

## 1. 需求背景

### 1.1 当前实现

当前算子在算子包中的 `orderIndex` 是串行唯一的（0, 1, 2, 3...），通过上移/下移按钮调整顺序。这种设计假设所有算子都是串行执行的。

### 1.2 问题分析

在实际业务场景中，某些算子需要并行执行，即它们的执行顺序可以相同。例如：

- **场景 1**：数据清洗算子需要同时处理多个数据源，这些算子可以并行执行
- **场景 2**：多个独立的特征提取算子可以同时执行，然后再汇总结果

当前设计无法支持这种并行执行场景。

### 1.3 需求目标

支持算子的 `orderIndex` 可以重复，相同 `orderIndex` 的算子表示可以并行执行。同时提供更灵活的 `orderIndex` 编辑方式，支持单个和批量编辑。

---

## 2. 核心设计思路

### 2.1 orderIndex 可重复

- `orderIndex` 不再强制唯一，可以重复
- 相同 `orderIndex` 的算子表示可以并行执行
- 不同 `orderIndex` 的算子按顺序执行

### 2.2 手动设置 orderIndex

- 添加算子时，用户手动输入 `orderIndex`
- 添加算子后，可以修改 `orderIndex`
- 支持单个算子的 orderIndex 编辑
- 支持批量编辑多个算子的 orderIndex

### 2.3 移除上移/下移功能

- 移除"上移"、"下移"按钮
- 用户直接通过编辑 orderIndex 调整顺序

### 2.4 展示排序规则

算子列表按以下规则排序：
1. **orderIndex 升序**（优先）
2. **创建时间降序**（orderIndex 相同时按创建时间倒序）

## 3. 功能需求（详细）

### 3.1 添加算子时设置 orderIndex

#### 功能描述

在添加算子到算子包时，用户可以手动输入 `orderIndex`。

#### 前端交互

1. 点击"添加算子"按钮，打开添加算子弹窗
2. 弹窗中包含以下字段：
   - 选择算子（下拉选择框）
   - 执行顺序（orderIndex，数字输入框，默认值为 `1`）
3. 用户输入 `orderIndex`，点击"确定"提交

#### 验证规则

- `orderIndex` 必须为正整数（>= 1）
- 如果用户未输入 `orderIndex`，默认值为 `1`

#### 后端实现

- 接收请求中的 `orderIndex`
- 如果 `orderIndex` 为 `null`，使用默认值 `1`
- 验证 `orderIndex` >= 1，否则抛出异常

---

### 3.2 编辑单个算子的 orderIndex

#### 功能描述

用户可以编辑单个算子的 `orderIndex`。

#### 前端交互

1. 在算子列表中，`orderIndex` 列显示当前值
2. 点击 `orderIndex` 值，弹出编辑弹窗
3. 在弹窗中输入新的 `orderIndex`
4. 点击"确定"保存

#### 验证规则

- `orderIndex` 必须为正整数（>= 1）

#### 后端实现

- 使用 `PUT /v1/packages/{id}/operators/{packageOperatorId}` 接口
- 更新 `PackageOperator` 的 `orderIndex` 字段
- 验证 `orderIndex` >= 1

---

### 3.3 批量编辑算子的 orderIndex

#### 功能描述

支持批量编辑多个算子的 `orderIndex`，用于快速设置一批算子的执行顺序。

#### 前端交互

1. 在算子列表中，勾选多个算子（复选框）
2. 点击"批量设置顺序"按钮
3. 弹出批量编辑弹窗，包含以下字段：
   - 执行顺序（orderIndex，数字输入框）
   - 选择要设置的算子列表（显示已勾选的算子）
4. 输入 `orderIndex`，点击"确定"保存
5. 所有勾选的算子的 `orderIndex` 都会被设置为相同值

#### 验证规则

- `orderIndex` 必须为正整数（>= 1）
- 至少勾选 1 个算子

#### 后端实现

- 新增接口：`POST /v1/packages/{id}/operators/batch-update-order`
- 请求体：`{ operatorIds: [1, 2, 3], orderIndex: 5 }`
- 批量更新指定算子的 `orderIndex` 为相同值
- 验证 `orderIndex` >= 1

---

### 3.4 算子列表展示优化

#### 功能描述

算子列表按 `orderIndex 升序 + 创建时间降序` 排序展示。

#### 前端实现

- 算子列表的 `orderIndex` 列显示当前值
- 列表按 `orderIndex ASC, createdAt DESC` 排序
- 显示操作列：启用/禁用、编辑、移除
- **移除**：上移/下移按钮

#### 后端实现

- 修改查询方法：`getPackageOperators`
- 排序规则：`orderIndex ASC, createdAt DESC`

---

### 3.5 移除上移/下移接口

#### 功能描述

移除上移/下移相关的 API 接口。

#### 后端修改

**删除的接口**：
- `POST /v1/packages/{id}/operators/{operatorId}/reorder`
- `POST /v1/packages/{id}/operators/reorder`

**删除的方法**：
- `reorderOperators(Long packageId, ReorderOperatorsRequest request, String username)`

#### 前端修改

**删除的 API 方法**：
- `reorderOperators(packageId, operatorId, direction)`
- `bulkReorderOperators(packageId, operators)`

**删除的 UI 元素**：
- 算子列表操作列的"上移"按钮
- 算子列表操作列的"下移"按钮
- `handleReorder` 处理函数

## 4. 数据模型设计

### 4.1 是否需要修改

**数据库表结构**：无需修改

- `package_operators` 表的 `order_index` 字段仍然是 `INTEGER` 类型
- 唯一约束仍然是 `(package_id, operator_id)`，不涉及 `order_index`
- 因此 `order_index` 可以重复

**实体类**：无需修改

- `PackageOperator` 实体类的 `orderIndex` 字段不变

### 4.2 字段验证

**PackageOperatorRequest DTO**：

| 字段 | 类型 | 默认值 | 验证规则 |
|------|------|--------|---------|
| `orderIndex` | `Integer` | `1` | `@Min(1, message = "orderIndex must be a positive integer")` |

## 5. API 接口设计

### 5.1 新增接口

#### 批量更新算子 orderIndex

**接口定义**：

```
POST /v1/packages/{id}/operators/batch-update-order
```

**请求参数**：

```json
{
  "operatorIds": [1, 2, 3],
  "orderIndex": 5
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `operatorIds` | `Long[]` | 是 | 要更新的算子 ID 列表 |
| `orderIndex` | `Integer` | 是 | 新的 orderIndex 值（必须 >= 1） |

**响应**：

```json
{
  "success": true,
  "message": "算子执行顺序批量更新成功",
  "data": null
}
```

---

### 5.2 修改接口

#### 添加算子到算子包

**接口定义**：

```
POST /v1/packages/{id}/operators
```

**修改内容**：

- `orderIndex` 如果为 `null`，默认值为 `1`
- 添加 `@Min(1)` 验证注解

#### 更新算子配置

**接口定义**：

```
PUT /v1/packages/{id}/operators/{packageOperatorId}
```

**修改内容**：

- `orderIndex` 如果为 `null`，不更新该字段
- 添加 `@Min(1)` 验证注解（如果传入 orderIndex）

#### 获取算子包的算子列表

**接口定义**：

```
GET /v1/packages/{id}/operators
```

**修改内容**：

- 排序规则：`orderIndex ASC, createdAt DESC`

### 5.3 删除接口

## 8. 风险与注意事项

### 8.1 风险分析

#### 风险 1：数据兼容性

**描述**：现有算子包中的算子 `orderIndex` 可能是 0、1、2...，不符合新的正整数要求（>= 1）。

**影响**：如果用户查询现有算子包，`orderIndex = 0` 的算子会在列表中展示。

**应对措施**：
- 在查询时自动过滤掉 `orderIndex < 1` 的算子（可选）
- 或者提供数据库迁移脚本，将所有 `orderIndex` 的值加 1（可选）
- **用户确认**：不考虑历史数据（当前项目无真实数据）

#### 风险 2：批量编辑的并发问题

**描述**：多个用户同时批量编辑同一批算子的 `orderIndex`，可能出现数据覆盖。

**影响**：后编辑的用户可能会覆盖前编辑用户的修改。

**应对措施**：
- 使用乐观锁（`@Version` 注解）
- 在接口中添加操作前验证

#### 风险 3：orderIndex 重复导致的执行逻辑

**描述**：算子包的执行逻辑需要支持相同 `orderIndex` 的算子并行执行。

**影响**：如果执行引擎不支持并行执行，会导致执行错误。

**应对措施**：
- 与执行引擎团队确认支持情况
- 在文档中说明并行执行的前提条件

---

### 8.2 注意事项

1. **前端验证**：`orderIndex` 必须为正整数（>= 1），在前端和后端都需要验证
2. **排序规则**：算子列表按 `orderIndex ASC, createdAt DESC` 排序
3. **批量操作**：批量编辑时，所有选中算子的 `orderIndex` 会被设置为相同值
4. **接口移除**：上移/下移接口删除后，前端需要同步移除相关代码
