# 算子包导入功能 - 需求设计文档

**文档版本**: v1.1
**创建日期**: 2026-02-23
**状态**: 已完成

---

## 1. 需求概述

### 1.1 需求背景

当前系统支持将算子包导出为 ZIP 压缩包，但不支持从 ZIP 文件导入算子包。用户希望实现导入功能，以便：

1. 从其他系统导入算子包
2. 从本地备份恢复算子包
3. 实现算子包的分享和迁移

### 1.2 需求目标

提供算子包导入功能，支持从 ZIP 压缩包导入算子包，具体目标：

- 解析 ZIP 压缩包，提取算子、公共库和元数据信息
- 智能复用现有算子和公共库（避免重复创建）
- 创建新的算子包，自动建立算子和公共库的关联关系
- 支持导入数据的完整性验证

### 1.3 原始需求记录

用户提出："为算子管理系统增加算子包导入的功能，导入的数据包结构就是当前我们打包导出的结构"

讨论要点记录：
- 导入策略：算子和公共库采用智能复用，算子包按新建处理
- 冲突处理：算子和公共库是共享资源，导入人要保证内容适用性；数据不完整不允许导入
- 导入入口：算子包列表页
- 导入预览：暂不实现预览，验证通过后直接导入
- API 设计：使用专门的导入接口 `POST /v1/packages/import`

## 2. 功能需求

### 2.1 功能描述

实现算子包导入功能，用户可以在算子包列表页上传 ZIP 压缩包，系统解析后自动创建或更新相关资源。

### 2.2 用户故事

**场景 1：导入新的算子包**
1. 用户在算子包列表页点击"导入算子包"按钮
2. 选择本地的 ZIP 压缩包文件
3. 系统验证数据完整性
4. 系统创建新算子包、更新或创建相关算子和公共库
5. 显示导入成功提示，刷新列表

**场景 2：导入已存在的算子包（名称冲突）**
1. 用户上传的算子包名称已存在
2. 系统自动为算子包名称添加数字后缀（如 `my_package_2`）
3. 其他资源按智能复用策略处理

### 2.3 功能清单

| 功能点 | 描述 | 优先级 |
|-------|------|-------|
| 导入入口 | 在算子包列表页添加"导入算子包"按钮 | P0 |
| ZIP 解析 | 解析 ZIP 压缩包，提取元数据和文件 | P0 |
| 完整性验证 | 验证元数据文件、算子代码文件、公共库文件的完整性 | P0 |
| 算子处理 | 智能复用或创建算子，更新基本信息和代码 | P0 |
| 公共库处理 | 智能复用或创建公共库，更新代码文件 | P0 |
| 算子包创建 | 创建新算子包，名称冲突自动加后缀 | P0 |
| 关联关系建立 | 建立 OperatorPackage、PackageOperator、PackageCommonLibrary 关联 | P0 |
| 结果反馈 | 显示导入结果统计（更新/创建数量） | P0 |
| 异常提示 | 数据不完整时给出明确的错误信息 | P0 |

## 3. 非功能需求

### 3.1 性能要求

- 文件上传支持最大 10MB 的 ZIP 文件
- 导入处理时间应在 10 秒内完成（正常大小的算子包）

### 3.2 安全要求

- 导入接口需要用户认证（JWT Token）
- 验证文件类型，只接受 ZIP 格式
- 防止路径遍历攻击（验证 ZIP 文件路径）

### 3.3 可用性要求

- 导入失败时，必须给出清晰的错误信息，指出具体问题（如缺少哪个文件）
- 导入成功后，自动刷新算子包列表，显示新导入的算子包

## 4. 技术方案

### 4.1 整体设计

#### 4.1.1 导入策略

**算子（Operator）- 智能复用 + 选择性更新**
- 存在：复用现有算子实例，更新基本信息（name, objectCode, dataFormat, generator 等）和代码内容，保留参数和业务逻辑
- 不存在：创建新算子

**公共库（CommonLibrary）- 智能复用 + 选择性更新**
- 存在：复用现有公共库，删除所有旧文件，替换为导入包中的文件（以导入的为准），保留其他信息
- 不存在：创建新公共库

**算子包（OperatorPackage）- 始终新建 + 名称冲突处理**
- 名称不冲突：直接创建
- 名称冲突：自动加数字后缀区分（`my_package`, `my_package_2`, `my_package_3`...）

#### 4.1.2 数据包结构（与导出一致）

```
{包名}/
├── {版本号}/
│   ├── models/                     # MODEL 类型的公共库
│   │   └── ${fileName}
│   └── operators/
│       ├── constants/              # CONSTANT 类型的公共库
│       │   └── ${fileName}
│       ├── groovy/                # 算子代码
│       │   └── ${operatorCode}.groovy
│       ├── method/                # METHOD 类型的公共库
│       │   └── ${fileName}
│       └── metainfo_operators.yml # 元数据文件（YAML）
```

#### 4.1.3 元数据文件格式

```yaml
businessName: {业务场景}
version: {版本号}
operators:
  instances:
    - operator_code: {算子编码}
      name: {算子名称}
      object_code: {对象编码}
      data_format: {数据格式}
      generator: {生成方式}
      order_no: {执行顺序}
```

### 4.2 数据库设计

本次功能不需要数据库结构变更，使用现有表结构：
- `operator_packages` - 算子包表
- `package_operators` - 算子包-算子关联表
- `package_common_libraries` - 算子包-公共库关联表
- `operators` - 算子表
- `common_libraries` - 公共库表
- `common_library_files` - 公共库文件表

### 4.3 接口设计

#### 4.3.1 导入接口

**接口路径**：`POST /v1/packages/import`

**认证**：需要（JWT Token）

**请求格式**：
```
Content-Type: multipart/form-data
```

**请求参数**：
```
file: File  // ZIP 压缩包文件
```

**响应格式（成功）**：
```json
{
  "success": true,
  "message": "算子包导入成功",
  "data": {
    "id": 123,
    "name": "my_package",
    "operatorsUpdated": 3,
    "operatorsCreated": 2,
    "librariesUpdated": 5,
    "librariesCreated": 1
  }
}
```

**响应格式（失败）**：
```json
{
  "success": false,
  "message": "导入失败：数据不完整",
  "error": "缺少算子代码文件：my_operator.groovy"
}
```

#### 4.3.2 异常处理

| 异常情况 | 响应码 | 响应消息 |
|---------|--------|---------|
| 文件格式错误 | 400 | 只支持 ZIP 格式文件 |
| 元数据文件缺失 | 400 | 导入包缺少元数据文件 metainfo_operators.yml |
| 元数据格式错误 | 400 | 元数据文件格式错误 |
| 算子代码文件缺失 | 400 | 缺少算子代码文件：{operatorCode}.groovy |
| 公共库文件为空 | 400 | 公共库 {libraryName} 没有文件 |
| 目录结构错误 | 400 | 导入包目录结构不符合预期 |
| 算子包创建失败 | 500 | 创建算子包失败 |

### 4.4 前端设计

#### 4.4.1 UI 入口

**位置**：`operator-manager-web/src/pages/package/list.tsx`

**布局**：
```
+-------------------------------------------+
|  算子包管理                [+ 创建算子包] [📥 导入算子包] |
+-------------------------------------------+
```

#### 4.4.2 交互流程

1. 用户点击"导入算子包"按钮

#### 4.4.4 API 调用

**operator-manager-web/src/api/package.ts**
```typescript
/**
 * 导入算子包
 * @param file ZIP 压缩包文件
 * @returns 导入结果
 */
export const importPackage = async (file: File): Promise<ApiResponse<PackageImportResponse>> => {
  const formData = new FormData();
  formData.append('file', file);

  return request.post('/v1/packages/import', formData, {
    headers: {
      'Content-Type': 'multipart/form-data',
    },
  });
};
```

#### 4.4.5 类型定义

**operator-manager-web/src/types/package.ts**
```typescript

 * 算子包导入响应

  librariesUpdated: number; // 更新的公共库数量
```

## 5. 实施计划

### 5.1 任务分解

#### 阶段一：后端开发

1. **创建 DTO 类**
   - `PackageImportResponse.java`
   - `PackageImportMetadata.java`
   - `PackageImportMetadata.OperatorMetadata.java`

2. **实现 PackageImportService**
   - 解析 ZIP 文件，提取元数据
   - 数据完整性验证
   - 处理算子（查询/更新/创建）
   - 处理公共库（查询/更新/创建）
   - 创建算子包（处理名称冲突）
   - 建立关联关系

3. **实现 PackageImportController**
   - `POST /v1/packages/import` 接口

4. **编写测试脚本**
   - 测试正常导入场景
   - 测试算子复用场景
   - 测试公共库复用场景
   - 测试名称冲突场景
   - 测试数据不完整场景

#### 阶段二：前端开发

1. **添加类型定义**
   - `PackageImportResponse` 类型

2. **实现 API 调用**
   - `importPackage` 函数

3. **实现 UI 组件**
   - 在列表页添加"导入算子包"按钮
   - 实现文件选择和上传功能
   - 添加成功/失败提示

#### 阶段三：联调测试

1. 后端 API 测试
2. 前端功能测试
3. 完整导入流程测试

### 5.2 依赖关系

```
阶段一（后端开发）
  ├─ DTO 类创建（无依赖）
  ├─ Service 实现（依赖 DTO 类）
  ├─ Controller 实现（依赖 Service）
  └─ 测试脚本（依赖 Controller）

阶段二（前端开发）
  ├─ 类型定义（无依赖）
  ├─ API 调用（依赖类型定义）
  └─ UI 组件（依赖 API 调用）

阶段三（联调测试）
  └─ 前后端联调（依赖阶段一和阶段二完成）
```

### 5.3 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
|-----|------|------|---------|
| 算子更新影响其他算子包 | 高 | 高 | 在文档中明确告知用户，算子和公共库是共享资源 |
| 元数据格式变化导致兼容性问题 | 中 | 低 | 严格遵循现有导出格式，文档明确格式要求 |
| ZIP 解析异常 | 中 | 中 | 添加详细的异常处理和错误提示 |
| 文件大小超出限制 | 低 | 低 | 前端限制文件大小，后端也做验证 |

## 6. 验收标准

### 6.1 功能验收

- [x] 在算子包列表页可以看到"导入算子包"按钮
- [x] 点击按钮可以选择 ZIP 文件上传
- [x] 上传后可以成功导入新的算子包
- [x] 算子编码存在时，能正确更新算子的基本信息和代码
- [x] 公共库名称存在时，能正确更新公共库的代码文件
- [x] 算子包名称冲突时，自动加数字后缀
- [x] 数据不完整时，能给出明确的错误提示
- [x] 导入成功后，算子包列表能自动刷新显示新算子包
- [x] Legacy 模板下，constants、method、models 文件夹正确合并为单独的库
- [x] 算子包和代码库的关联关系正确建立（去重）

### 6.2 测试用例

#### 用例 1：正常导入新算子包
**前置条件**：系统中无任何算子、公共库、算子包
**操作步骤**：
1. 上传完整的算子包 ZIP 文件
**预期结果**：
- 导入成功
- 创建新的算子包
- 创建所有算子
- 创建所有公共库
- 建立正确的关联关系

#### 用例 2：导入算子包（算子已存在）
**前置条件**：系统中已存在 `operator_code='test_operator'` 的算子
**操作步骤**：
1. 上传包含 `test_operator` 的算子包 ZIP 文件
**预期结果**：
- 导入成功
- 复用现有算子
- 更新算子的基本信息和代码
- 保留算子的参数和业务逻辑

#### 用例 3：导入算子包（公共库已存在）
**前置条件**：系统中已存在 `library_name='DateUtils'` 的公共库
**操作步骤**：
1. 上传包含 `DateUtils` 的算子包 ZIP 文件
**预期结果**：
- 导入成功
- 复用现有公共库
- 删除旧文件，替换为新文件
- 保留公共库的其他信息

#### 用例 4：导入算子包（名称冲突）
**前置条件**：系统中已存在 `name='my_package'` 的算子包
**操作步骤**：
1. 上传包名为 `my_package` 的 ZIP 文件
**预期结果**：
- 导入成功
- 新建算子包，名称为 `my_package_2`

#### 用例 5：数据不完整
**操作步骤**：
1. 上传缺少算子代码文件的 ZIP 文件
**预期结果**：
- 导入失败
- 显示错误提示："缺少算子代码文件：xxx.groovy"
- 不创建任何资源

#### 用例 6：文件格式错误
**操作步骤**：
1. 上传非 ZIP 格式的文件（如 .txt）
**预期结果**：
- 导入失败
- 显示错误提示："只支持 ZIP 格式文件"
